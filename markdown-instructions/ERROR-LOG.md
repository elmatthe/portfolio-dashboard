## Currency Conversion Findings

🔴 **CRITICAL — #FX-1 — FX rate never populated on imported transactions.**
After uploading Questrade + Wealthsimple, every USD transaction in the DB has `fx_rate_to_cad: null` and `net_cad: null`. Probed via `GET /api/data/export-json`:
```
2024-09-05  BUY  MSFT   fx_rate_to_cad=null  net_cad=null
2024-09-05  BUY  AMZN   fx_rate_to_cad=null  net_cad=null
2025-02-05  BUY  GOOGL  fx_rate_to_cad=null  net_cad=null
2025-03-05  BUY  MSFT   fx_rate_to_cad=null  net_cad=null
... (all 21 USD rows blank)
```
This **directly contradicts** the CHANGELOG 0.3.0 claim:
> "FX rate at trade date, not today's rate ... Each Transaction stores its own `fx_rate_to_cad` and `net_cad` at the moment of the trade."

Root cause: `backend/parser.py` constructs every `Transaction(...)` without calling `FXService.populate_transaction()`. The parser-registry-driven Questrade and Wealthsimple paths skip FX entirely. `backend/fx/rates.py::FXService.populate_transaction()` exists (lines 140–145) but is never invoked.

🔴 **CRITICAL — #FX-2 — Capital-gains aggregator sums CAD and USD without conversion.**
`backend/acb.py::compute()` keeps ACB in the security's native currency (documented at lines 15–17). It then aggregates `total_taxable_gain` (line 266 of `acb.py`) by **summing `g.total_gain`** for taxable gains — across all currencies indiscriminately. With the current fixtures:
- 7 CAD-denominated taxable gains: -30.42, -48.25, 146.61, 30.37, +130.80 = +229.10 CAD
- 2 USD-denominated taxable gains: +26.66 USD, -10.20 USD = +16.46 USD

The API reports `total_taxable_gain: 245.56` — i.e. it added the raw numbers and lost the unit. At the audit-time spot rate (~1.377 USD/CAD), the *correct* CAD figure would be ≈ 229.10 + 16.46×1.377 = **+251.76 CAD**. The reported figure is wrong by ~$6.20 on this dataset, would grow with USD exposure, and is what would be displayed on the CRA Tax Report PDF. **For a tool that brands itself "CRA-compliant" this is a tax-reporting bug, not a display nit.**

🟡 **#FX-3 — `FXService.STATIC_RATES_TO_CAD` are stale 2024 spot rates** (USD=1.36, GBP=1.72, etc., set in `backend/fx/rates.py` lines 38–48). When `FX_LIVE_RATES` is off (the default), every conversion uses these constants regardless of date. A 2020 USD trade is converted at the same rate as a 2024 USD trade. The CHANGELOG calls this out as the offline fallback, but there is no warning in the UI or response payload when the static fallback is in effect.

🟡 **#FX-4 — `/api/holdings?currency=...` is not a parameter.** The audit instructions assume a `currency` query param on `/api/holdings`. The OpenAPI shows `account` and `period` only. Unknown query params are silently ignored by FastAPI, so probing `?currency=usd` returns identical data to `?currency=cad`. There is no per-request CAD/USD/Combined toggle on the backend; currency presentation appears to be handled entirely on the frontend by reading individual holding fields.

## TFSA Room Verification

Query: `GET /api/tfsa/room` (settings: `tfsa_birth_year=null`, `tfsa_resident_since=null`).

Expected cumulative limit 2009→2026 for a Canadian who turned 18 before 2009:
```
2009-2012: 5000×4 = 20,000
2013-2014: 5500×2 = 11,000

2016-2018: 5500×3 = 16,500
2019-2022: 6000×4 = 24,000

2024-2026: 7000×3 = 21,000

```
**Actual from API:** `total_room_accumulated: 109,000` ✓ matches.

(Note: the audit instructions list **$95,000** "by end of 2025" — that figure is incorrect. The correct 2009-2025 cumulative is **$102,000**; the API's $109k includes 2026's $7k limit.)

🟡 **#TFSA-1 — Silent default of `birth_year=None` to eligibility=2009.**
`backend/tfsa.py` line 94: `age18_year = (birth_year + 18) if birth_year else 2009`. When the user hasn't filled in their birth year (the default settings state), the calculator silently assumes they have been eligible since 2009 and reports the full $109k of room. The response *does* include `missing_settings: ["tfsa_birth_year", "tfsa_resident_since"]` (line 88-91), but `eligibility_start_year` is returned as `2009` without an "estimated" flag. A 25-year-old user with no settings filled in will see "$109,000 of room available" — a six-figure overstatement.

🟡 **#TFSA-2 — `total_room_accumulated` conflates limits with withdrawal credits.**
Line 157: `total_room_accumulated = total_limit_accumulated + withdraw_prior_years`. The field's name implies "the CRA sum" but the value also folds in any prior-year withdrawals. Two distinct numbers (annual-room sum, withdrawals-restored-next-year) get smushed into one. The annual breakdown rows still surface limits per year, but the headline number is misleading.

🟡 **#TFSA-3 — USD TFSA contributions converted at 1:1.**
Line 104-110: When `tx.currency == "USD"`, the code reads `amt = amt * 1.0` (a no-op). The comment acknowledges "the exact CRA rule is to use the BoC rate on the contribution date — surface that as a TODO via the missing-settings note" — but no such note is ever added. USD contributions are silently undervalued / not properly converted.

🟡 **#TFSA-4 — Birth year not bounds-checked.**
No check that `birth_year` is realistic (e.g. > 1900, < current year - 17). A `birth_year=3000` produces `age18_year=3018`, which makes `eligibility_start=max(3018, ..., 2009) > current_year`, leading to an empty annual_breakdown loop. Returns 0 room, no error. A `birth_year=2026` (current year) produces eligibility=2044 — silently 0 room with no validation warning.

Accounts 29171399 and 29461802 are both Margin in the same broker — when the user transfers positions between them, the source account drops 92% and the destination jumps 360%. The combined number is fine because the per-account distortions cancel. **For any user viewing per-account performance on the dashboard, the percentages are nonsense whenever a transfer between accounts occurred.** Root cause in `backend/portfolio.py` lines 274-294: `net_dep_in_period` only sums DEPOSIT/CONTRIBUTION/WITHDRAWAL actions, not inter-account transfers (the TRANSFER action exists in the `Action` Literal but isn't included in the netting set).

🔴 **#PORT-2 — `period=all` returns 0% return.**
The combined row for `period=all` always has `period_start_value_cad=0.0, period_return_cad=0.0, period_return_pct=0.0`. The chosen denominator-rule (line 291: `denom = period_start_value if > 0 else net_dep_in_period`) yields 0/0 for "all". This is the most natural period to show users their lifetime ROI — and the dashboard returns zero. Functional gap.

🟡 **#PORT-3 — Unknown `period` value silently falls through to "all".**
`?period=garbage` returns the same payload as `?period=all` with `period_label: "all"`. No 422 / validation hint. The query parameter is `period: str | None`, so FastAPI accepts any string. Adds confusion when users mistype.

🟡 **#PORT-4 — Unknown `account` value silently returns ALL holdings.**
`?account=RRSP` returns 7 holdings (the full set) even though the user has no RRSP account. `?account=bogus` likewise returns 7. The intent (from the helper `_filter_holdings(account)` in `portfolio.py`) is for unknown values to no-op, but a user filtering by RRSP will see Margin/TFSA holdings under the "RRSP" tab.

🟡 **#PORT-5 — `account=29171399` (account number) works but holdings response lacks account_number.**
The holdings response only includes `account_type`, not `account_number`. When two accounts share an account_type (e.g., two Margin accounts), the consumer can't tell which row is which until they probe `/api/portfolio` for the account list.

## Rebalancer Verification

Test scenario A — new-money mode, $5000 across VEQT.TO/TFSA, VOO/Margin, AAPL/Margin (50/30/20):

```
new_money_cad = 5000
portfolio_value_cad = 14369.43

Actions returned:
```

🔴 **#REB-1 — `mode=new_money` produces buys totalling 73% MORE than the available new money.**
`backend/rebalance.py` lines 65, 85-90: the algorithm computes each target's `delta_cad` against the **post-injection pool** (`total_holdings_cad + new_money_cad = $19,369`), then floors to whole shares. There is no constraint that the sum of `cost_cad` for all BUY actions stay ≤ `new_money_cad`. A user following these instructions would need an additional $3,650 of cash they didn't budget for.

Test scenario B — rebalance mode, same targets, zero new money:

```
Actions returned:
  SELL VEQT.TO TFSA  24 sh × 59.19  → 1420.56 CAD
  BUY  VOO     Margin 3 sh × 682.84 → 2821.98 CAD
  BUY  AAPL    Margin 4 sh × 304.99 → 1680.58 CAD
  Resulting %s: 50.25 / 26.19 / 17.54  (= 93.98%, not 100%)
```
🔴 **#REB-2 — `mode=rebalance` produces buys ($4,502) exceeding sells ($1,421), leaving the user with a $3,081 cash shortfall.** No cash-neutrality constraint.

🔴 **#REB-3 — Sells in TFSA are funding buys in Margin (cross-account proceeds).** The rebalancer doesn't acknowledge per-account separation. A user can't legally fund a Margin purchase with TFSA proceeds without first triggering a TFSA withdrawal — the recommendation is unactionable as-stated.

🟡 **#REB-4 — Targets sum mis-display when interpreted in [0,1] units.**
If the user submits `target_pct` as a fraction (0.5, 0.3, 0.2) summing to 1.0, the validator (line 33-42) reports "Target percentages sum to **1.00%** — they must sum to 100% before rebalancing." The message embeds the unit "%" assuming the user submitted percentages, but the implication ("1.00%" reads as "1/100 of a percent") is misleading. Either accept either convention, or normalize and document the expected scale.

🟡 **#REB-5 — `portfolio_value_cad: 0.0` is returned on the validation failure path** (line 41) even though the actual portfolio is non-empty. The user's UI may show a $0 portfolio when really the only problem was an off-by-decimal-point in targets.

🟡 **#REB-6 — New positions (not currently held) are rejected with a warning** rather than being supported (line 75 in `rebalance.py`). The CHANGELOG/README implies "set target percentages, generate buy/sell instructions" — implying you can establish new positions. Audit-instructed expectation §3.7 #3 ("Tickers not currently held can still be targeted") also assumes this works. It does not.

🟡 **#REB-7 — The cap-gains tax warning is hardcoded to Margin only** (line 123): `any(a.action == "SELL" and a.account_type == "Margin")`. The Account-type list now includes `Non-Registered`, `Individual`, `IRA`, `Roth IRA`, `Traditional IRA`, `LIRA`, `FHSA`, `Other` — all of which can be taxable. Sells in Non-Reg / Individual will trigger gains without the warning.

🟡 **#REB-8 — Negative `new_money_cad` is rejected but the rebalance is not redirected to `mode=rebalance`.** A "withdraw $X to rebalance" scenario isn't supported — the audit instructions §3.7 explicitly call this out as a desirable behavior (graceful handle of withdrawal scenarios).

## ACB / Capital Gains — Additional Engine Findings

🟡 **#ACB-1 — `fx_rate_for_date` parameter declared and unused.**
`backend/acb.py::compute()` line 82 accepts `fx_rate_for_date: Callable[[date, str], float] | None`. It is never invoked within the function. Dead parameter — either remove it or wire it up so CAD aggregates can be produced.

🟡 **#ACB-2 — Superficial-loss day-of-sale exclusion likely wrong.**
Line 186: `qualifying_buys = [(d,q) for d,q in all_buys if window_start <= d <= window_end and d != entry.transaction_date]`. The `d != entry.transaction_date` clause excludes same-day repurchases from triggering denial. CRA's 30-day window is **inclusive** of the sale day — a same-day buy is still a superficial loss trigger. Will under-report denials.

🟡 **#ACB-3 — SPLIT handler has the wrong abstraction.**
Lines 163-168:
```
if tx.quantity and ledger.total_shares > 0:
    ratio_target = (ledger.total_shares + tx.quantity) / ledger.total_shares
    ledger.total_shares *= ratio_target
```
This is algebraically `total_shares + tx.quantity`, then `total_cost` is left unchanged (which is correct for splits). But the docstring at line 165 says "Quantity carries the new share count delta" — yet the calculation `total_shares *= ratio_target` is equivalent to multiplying by 1 + delta/old, which is correct *only if* quantity is the delta and not the new count. Comment-vs-code ambiguity, plus no SPLIT examples in the test fixtures, makes this UNTESTABLE.

🟡 **#ACB-4 — `_KNOWN_DESCRIPTION_TICKERS` hand-coded list is short.**
`backend/parser.py` lines 261-270 lists only 8 hand-mapped patterns (AAPL, VOO, MSFT, NVDA, AMZN, GOOGL/META/TSLA). Any Questrade internal ID (e.g., `A603109`, `V007563`) that isn't one of these falls through to a yfinance.search() call, which the parser warns is unreliable. Coverage gap.

## Electron / Desktop Findings

🔴 **#ELEC-1 — Backend process killed with TerminateProcess on Windows.**
`electron/main.js` lines 282-287:
```
if (process.platform === "win32") {
  backendProcess.kill();
} else {
  backendProcess.kill("SIGTERM");
}
```
On Windows, `ChildProcess.kill()` with no signal sends a TerminateProcess equivalent — no graceful shutdown signal that Python can catch. SQLite WAL files are not checkpointed; the FastAPI lifespan shutdown handler is skipped. Evidence in `%APPDATA%\Portfolio Dashboard\profiles\bc1bcebe\portfolio.db-wal` is **4.5 MB** (much larger than expected after a clean shutdown). This compounds with #DB-1 below.

🟡 **#ELEC-2 — `shell.openExternal(url)` is invoked without scheme validation.**
`electron/main.js` line 248: every popup-window-open URL is forwarded to `shell.openExternal(url)`. No allow-list of `http(s):` only — a renderer-controlled `file://` or other custom-scheme URL would still be passed to the OS shell. Low risk because the renderer loads only local content, but a defensive check is cheap.

🟡 **#ELEC-3 — No Content-Security-Policy set on the main BrowserWindow.**
`webPreferences` (lines 225-230) enables `contextIsolation` and disables `nodeIntegration` (good), but no CSP is attached. In production the window loads from `file://` with no CSP header. If a malicious payload ever reached the renderer (e.g. via a tampered PDF/XLSX rendered in the future), it would have full network access.

🟡 **#ELEC-4 — Main window's `sandbox: false`** while the splash window has `sandbox: true`. Inconsistent; the main window should also be sandboxed where possible.

🟡 **#ELEC-5 — `dev` script in `electron/package.json` line 11 uses bash syntax on Windows.**
`"dev": "NODE_ENV=development electron ."` — fails on Windows PowerShell. Use `cross-env` or split via npm scripts.

🟡 **#ELEC-6 — `README.md` developer doc repeats the same Bash-only syntax** (line 272: `NODE_ENV=development npm start`). The README claims Windows support is the primary target but the developer instructions assume a POSIX shell.

🟡 **#ELEC-7 — No `app.setName()` / no signed builds** — Windows SmartScreen will flag the installer (acknowledged in README line 110-112). Not a code bug, but a portfolio-readiness item.

🟡 **#ELEC-8 — `appId: com.yourname.portfoliodashboard`** is a placeholder reverse-DNS. Replace before any signing/publish.

🟡 **#ELEC-9 — Missing app icons.**
